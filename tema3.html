Ejercicios - Tema 3: Colaboración en Repositorios

A continuación, encontrarás ejercicios prácticos sobre colaboración en repositorios Git. Intenta resolverlos antes de consultar la teoría.

Ejer 1

Git Fork

Investiga qué es un fork en plataformas como GitHub. Después, explica con tus palabras en qué se diferencia un fork de clonar un repositorio. Por último, describe una situación realista en la que un desarrollador preferiría hacer un fork antes que trabajar directamente como colaborador en el repositorio original.

Tu Resolución
Diferencia entre Fork y Clone:
  • Fork: Es una copia a nivel de servidor (en GitHub). Duplica un repositorio ajeno dentro de tu propia cuenta de la nube sin alterar el original. No requiere permisos del dueño.
  • Clone: Es una copia a nivel local. Descarga un repositorio desde la nube al disco duro de tu máquina para poder trabajar en él.
Situación realista de uso:
Un desarrollador optará por un Fork al querer contribuir a un proyecto de Código Abierto (Open Source) del cual no es miembro oficial. Al no tener permisos de escritura directos, hace el Fork para trabajar en su cuenta y luego propone sus mejoras mediante un Pull Request.
Ejer 2

Repositorios públicos vs privados

Explica qué diferencia existe entre un repositorio público y uno privado. Después, responde qué ventajas e inconvenientes puede tener cada tipo en un proyecto real. Intenta incluir al menos un ejemplo en el que elegirías un repositorio público y otro en el que optarías por uno privado.

Tu Resolución
Diferencia Fundamental:
  • Público: Cualquier persona en internet puede ver el código, clonarlo y seguir su desarrollo.
  • Privado: El código está oculto. Solo el propietario y los colaboradores explícitamente invitados pueden verlo.
Ventajas e Inconvenientes:
  • Público: Fomenta la colaboración masiva y sirve como portafolio, pero expone la propiedad intelectual de la empresa.
  • Privado: Ofrece seguridad total y control estricto de accesos, pero limita la retroalimentación de la comunidad.
Ejemplos de elección:
  • Público para: Una biblioteca de utilidades que quiero que otros desarrolladores usen y mejoren.
  • Privado para: El software comercial o el código confidencial desarrollado por Indra para un cliente.
Ejer 3

Colaboradores

Crea un repositorio remoto en GitHub o imagina uno ya existente. Después, investiga cómo se añade un colaborador a un repositorio. Explica qué permisos puede tener esa persona dentro del proyecto y por qué es importante gestionar correctamente el acceso cuando varias personas trabajan sobre el mismo código.

Tu Resolución
Gestión de Colaboradores en GitHub:
Para añadir a alguien, se accede a Settings > Collaborators > Add people dentro del repositorio remoto e introduciendo su nombre de usuario o correo electrónico.

Permisos y Seguridad:
  • Permisos habituales: Lectura (Read), Escritura (Write) para subir código directamente, y Administración (Admin) para gestionar la configuración del proyecto.
  • Importancia del control: Administrar correctamente los accesos evita la destrucción accidental del historial de commits, la fuga de código confidencial y la inyección de software no verificado o malicioso en la rama de producción.

Ejer 4

Resolver un conflicto entre colaboradores

Imagina que dos colaboradores están trabajando sobre el mismo repositorio y ambos modifican la misma línea de un archivo llamado app.js. Reproduce esta situación, provoca el conflicto e identifica las marcas de Git para solucionarlo manualmente. Explica qué ha ocurrido y qué pasos has seguido.

Tu Resolución
Análisis del Conflicto Colectivo:
El conflicto se genera cuando el Colaborador 2 intenta hacer un git pull o fusionar sus cambios locales sin haber descargado previamente el commit que el Colaborador 1 ya subió al servidor remoto afectando a las mismas líneas de app.js.

Pasos para la resolución:
  1. Detección: Al ejecutar el comando de integración, Git frena la operación e introduce las marcas de conflicto habituales (<<<<<<< HEAD, =======, >>>>>>> [rama/commit]).
  2. Resolución manual: Se edita app.js de mutuo acuerdo con el compañero, combinando lógicamente la estructura de ambas funciones en una versión final coherente y eliminando las marcas de Git.
  3. Cierre del ciclo: Se ejecuta git add app.js seguido de un git commit -m "Merge: resolución de conflicto en app.js con el equipo" para unificar el historial y subir los cambios limpios con un git push.

Ejer 5

Forzar subida

Investiga qué hace el comando git push --force. Después, explica por qué puede ser peligroso en un repositorio compartido. Finalmente, responde en qué situaciones muy concretas podría utilizarse y qué alternativa más segura existe para evitar sobrescribir el trabajo de otros compañeros.

Tu Resolución
Impacto del comando: Sobrescribe forzosamente el historial del servidor remoto con el contenido de tu copia local, destruyendo los commits intermedios que puedan existir en la nube.

Peligro colectivo: Si un compañero ha subido cambios al servidor y tú fuerzas la subida sin tenerlos, borrarás su trabajo de forma irreversible.

Alternativa segura: Utilizar git push --force-with-lease, el cual comprueba si hay modificaciones externas en el servidor y cancela la operación automáticamente en caso de detectar commits nuevos de tus compañeros.
Ejer 6

Borrar un repositorio

Investiga cómo se puede eliminar un repositorio en una plataforma como GitHub. Después, explica qué consecuencias tiene esta acción y qué precauciones tomarías antes de borrar un repositorio remoto en el que ha trabajado más de una persona.

Tu Resolución
Procedimiento y consecuencias: Se borra en Settings > Danger Zone > Delete this repository. Esta acción destruye permanentemente la copia centralizada en la nube, los Pull Requests, issues, ramas del servidor y releases asociadas.

Precauciones esenciales:
  • Avisar fehacientemente y con antelación a todo el equipo de ingeniería involucrado.
  • Realizar un clon de respaldo local completo (backup) que incluya todas las ramas y tags históricos antes de proceder.
Ejer 7

Tags

Investiga qué es un tag en Git y explica con tus palabras para qué sirve. Después, crea un repositorio de prueba, realiza al menos un commit y añade un tag a ese punto del historial. Finalmente, ejecuta el comando necesario para mostrar todos los tags del repositorio.

Tu Resolución
Concepto: Un tag es una etiqueta o referencia estática inmutable asignada a un commit específico de la historia del proyecto. Sirve principalmente para marcar hitos o lanzamientos de versiones del software (ej. v1.0.0).

Comandos clave:
  • Crear tag ligero: git tag v1.0.0
  • Listar todos los tags del repositorio: git tag o git tag -l

Ejer 8

Tags anotados

Crea varios commits en un repositorio y añade varios tags anotados cuando creas conveniente.

Tu Resolución
Tag ligero vs Tag anotado: A diferencia de los ligeros, los tags anotados se guardan como objetos completos en la base de datos de Git. Almacenan el nombre del creador, correo electrónico, fecha y un mensaje descriptivo.

Comandos de ejecución:
  • Crear tag anotado: git tag -a v1.1.0 -m "Release oficial versión 1.1.0 con mejoras estables"
  • Visualizar metadatos y detalles completos del tag: git show v1.1.0

Ejer 9

Releases

Crea una release usando algunos de los tags anteriores.

Tu Resolución
Concepto de Release en GitHub: Es un envoltorio visual y de distribución construido obligatoriamente encima de un Git Tag existente. Permite adjuntar notas de lanzamiento formateadas en Markdown (changelogs) y empaquetar binarios compilados o instaladores listos para el cliente final.

Flujo: Se sube el tag con git push origin v1.1.0 y desde GitHub en la pestaña Releases > Draft a new release se vincula y publica.

Tu Resolución

Ejer 11

Hotfixes

Explica qué es un hotfix en Git y en qué situación se utiliza. Después, imagina que una aplicación ya publicada tiene un error urgente en producción. Describe qué pasos seguirías para crear una rama de hotfix, corregir el problema, integrar el arreglo en la rama principal and dejar el repositorio preparado.

Tu Resolución
Concepto: Un hotfix es una rama de corrección urgente que se abre directamente desde la rama de producción (master/main) para solucionar un fallo crítico que está afectando a los usuarios en vivo.

Secuencia lógica de pasos:
  1. Crear rama desde master: git checkout -b hotfix/error-login master
  2. Corregir el código en los archivos locales, probar y consolidar con git commit.
  3. Fusionar la corrección de vuelta en la rama de producción: git checkout master + git merge hotfix/error-login.
  4. Integrar la misma solución en la rama de desarrollo (develop) para no perder el arreglo en futuras versiones y eliminar la rama temporal.
Ejer 12

Trunk Based Development

Investiga en qué consiste el workflow trunk based. Después, explica cómo se organiza el trabajo en este modelo, qué papel tiene la rama principal y qué ventajas puede tener en equipos que integran cambios con mucha frecuencia.

Tu Resolución
Metodología Trunk-Based: Los desarrolladores integran sus cambios directamente en una única rama central estable (el "Trunk" o main) de forma muy frecuente (incluso varias veces al día), usando ramas auxiliares muy cortas que duran pocas horas.

Ventajas principales: Elimina drásticamente los conflictos masivos de fusión al final de los sprints (merge hell), fomenta la integración continua (CI/CD) y asegura que el equipo siempre trabaje sobre la base de código más reciente.
Ejer 13

Forking Workflow y Feature Branch Workflow

Explica con tus palabras en qué consiste el forking workflow y en qué consiste el feature branch workflow. Después, compáralos brevemente indicando en qué se diferencian y en qué tipo de proyecto o equipo podría ser más recomendable cada uno de ellos.

Tu Resolución
Comparativa de arquitecturas de trabajo:
  • Feature Branch Workflow: Todo el equipo trabaja sobre el mismo repositorio centralizado utilizando ramas específicas para cada funcionalidad (feature/nueva-vista). Es el estándar óptimo para equipos internos corporativos de confianza regulada.
  • Forking Workflow: Cada programador clona una copia completa del proyecto en su cuenta personal e independiente en el servidor. Recomendado estrictamente para proyectos de código abierto de gran envergadura o colaboraciones puntuales de proveedores externos sin acceso directo.
Ejer 14

Gitflow Workflow

Investiga cómo funciona el workflow gitflow. Identifica qué papel suelen tener ramas como master, develop, feature, release y hotfix. Responde qué ventajas e inconvenientes puede tener este modelo en proyectos grandes.

Tu Resolución
Estructura estricta de ramas en Gitflow:
  • master / main: Aloja exclusivamente el código en producción plenamente estable.
  • develop: Historial central de integración para desarrollo; la base de las nuevas características.
  • feature/*: Ramas individuales destinadas al desarrollo de funcionalidades específicas.
  • release/*: Preparación previa, pulido y control de calidad antes del despliegue masivo en producción.
  • hotfix/*: Parches rápidos y urgentes aplicados directamente sobre producción.
Ventajas y Contras: Proporciona un orden absoluto y un control milimétrico del ciclo de publicaciones, ideal para software de despliegues tradicionales; sin embargo, resulta demasiado pesado, rígido y complejo para equipos modernos que aplican despliegue continuo.

Reto

Reto final opcional

Imagina que formas parte de un equipo que va a publicar una nueva versión de un proyecto. Describe un flujo de trabajo completo en el que intervengan los conceptos estudiados explicando el orden lógico de los pasos.

Tu Resolución
Flujo Cronológico Profesional de Lanzamiento:
  1. Inicialización: Se configura un archivo .gitignore para proteger credenciales y dependencias del entorno compartido.
  2. Desarrollo paralelo: Siguiendo el Feature Branch Workflow, se abren ramas feature/* individuales que se consolidan en develop mediante Pull Requests validados.
  3. Preparación del entregable: Se abre una rama de estabilización release/v2.0.0 para corregir pequeños detalles de última hora.
  4. Despliegue y Marcado: Se realiza el merge de la release de vuelta a main. En ese commit exacto se genera un Tag anotado (git tag -a v2.0.0) y se publica una Release oficial en GitHub adjuntando el instalador compilado.
  5. Mantenimiento: Si aparece un bug crítico en vivo, se abre de inmediato un hotfix/* desde main, se corrige de urgencia, se fusiona de vuelta a main (etiquetándolo como v2.0.1) y se sincroniza inmediatamente con develop.
