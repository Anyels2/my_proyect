Ejercicios - Tema 4: Hooks, GitLab, Restore, Revert, Pipelines y GitHub Actions

A continuación, encontrarás ejercicios prácticos sobre automatización de tareas y control de cambios en entornos profesionales.

Ejer 1

Hook pre-commit y commit-msg

Investiga qué es un hook en Git. Después, explica qué función tienen concretamente los hooks pre-commit y commit-msg. Finalmente, describe un ejemplo realista de uso.

Tu Resolución
Concepto de Git Hook: Son scripts automatizados que Git ejecuta de forma nativa antes o después de eventos clave (como commits, pushes o merges) para validar flujos de trabajo.

Tipos específicos:
  • pre-commit: Se ejecuta antes de procesar el commit. Se usa para analizar el código de forma automática (pasando un Linter o formateador) garantizando que no se suban archivos con errores de sintaxis o malas prácticas.
  • commit-msg: Se ejecuta inmediatamente después de escribir el mensaje. Evalúa que el texto cumpla con un estándar específico, por ejemplo, verificar que empiece con el identificador de la tarea de Jira (ej. [INDRA-123] Fix login error).
Ejer 2

Hook pre-push

Explica qué hace el hook pre-push y en qué momento se ejecuta. Describe cómo podría utilizarse para impedir una subida incorrecta si los tests fallan.

Tu Resolución
Mecanismo pre-push: Se activa justo al ejecutar git push, antes de que los objetos se envíen de manera definitiva al servidor remoto.

Aplicación práctica: El script ejecuta la suite de pruebas automatizadas locales del proyecto (ej. npm test). Si alguna prueba unitaria falla, el script devuelve un código de salida distinto de cero (exit 1), lo que aborta automáticamente la subida. Esto actúa como un escudo de seguridad para que ningún desarrollador rompa la rama compartida por descuido.

Ejer 5

Crear una rama y hacer un Merge Request

Crea una rama, realiza un cambio y súbelo. Genera un Merge Request desde GitLab. Explica para qué sirve y su utilidad.

Tu Resolución
Concepto de Merge Request (MR): Es el equivalente en GitLab al Pull Request de GitHub. Es una solicitud formal para fusionar el código de una rama secundaria dentro de la rama principal.

Utilidad colaborativa: Funciona como un foro de revisión técnica. Permite a otros miembros del equipo examinar el código línea por línea, dejar comentarios de mejora, debatir soluciones y ejecutar análisis automatizados de calidad antes de incorporar el trabajo al código definitivo del cliente.

Ejer 6

Roles en GitLab y gestión de miembros

Investiga cómo se gestionan los miembros en GitLab. Explica la función de los roles y describe al menos dos diferentes.

Tu Resolución
Propósito del control de accesos: Garantiza el principio de mínimo privilegio en ingeniería de software. No todos los miembros deben poseer permisos idénticos para blindar las ramas críticas de producción frente a borrados o escrituras no autorizadas.

Descripción de Roles comunes:
  • Developer: Rol operativo estándar. Posee permisos de lectura y escritura completos para crear ramas secundarias y subir código, pero tiene prohibido aprobar Merge Requests protegidos o modificar la configuración estructural del proyecto.
  • Maintainer: Administrador técnico. Tiene autoridad absoluta para aceptar fusiones en ramas protegidas (como main), configurar variables de entorno del servidor y coordinar los lanzamientos de producción.
Ejer 7

git restore file

Modifica un archivo bajo control sin hacer commit. Usa git restore nombre-del-archivo para deshacer los cambios.

Tu Resolución
Comando ejecutado: git restore index.html

Efecto y comportamiento técnico: El comando descarta de forma fulminante e irreversible las modificaciones que se encontraban en el directorio de trabajo actual. Git sobreescribe el archivo local trayendo la última versión exacta que tenía guardada en el historial del último commit, dejando el comando git status limpio.
Ejer 8

git restore --staged file

Modifica un archivo, añádelo con add y sácalo con git restore --staged. Explica la diferencia técnica.

Tu Resolución
Mecánica del flag --staged: Saca el archivo modificado del área de preparación (Staging Area) y lo devuelve al estado de cambios sin preparar (Working Directory).

Diferencia fundamental:
  • git restore archivo: Destruye el código modificado de tus archivos físicos de forma permanente.
  • git restore --staged archivo: **Conserva intactas tus líneas de código**, solo deshace la acción del git add para que no entre en el próximo commit de manera inmediata.
Ejer 9

git revert <commit-hash>

Elige un commit antiguo y reviértelo utilizando su identificador hash. Comprueba el historial resultante.

Tu Resolución
Comando aplicado: git revert a1b2c3d4

Comportamiento del historial: A diferencia de comandos destructivos como git reset, revert **respeta el pasado**. Git calcula el cambio inverso al commit especificado y genera automáticamente un **nuevo commit adicional** que anula los cambios perjudiciales. De esta forma, el historial se mantiene intacto y transparente para el resto de desarrolladores remotos.
Ejer 10

GitLab Revert

Investiga la opción de revertir desde la interfaz web de GitLab. ¿En qué escenarios es útil y qué precauciones se deben contemplar?

Tu Resolución
Escenario de uso óptimo: Es idóneo cuando un commit integrado recientemente a través de un Merge Request rompe el entorno de producción de forma imprevista. Cualquier responsable técnico puede presionar el botón "Revert" directamente en la web de GitLab para restaurar la estabilidad de inmediato sin necesidad de clonar o encender la terminal local.

Precaución crítica: Al generar un nuevo commit en el servidor remoto, se debe avisar de inmediato a los compañeros del equipo de ingeniería para que ejecuten un git pull obligatorio antes de continuar programando, evitando así desajustes de versiones en local.
Ejer 11

GitLab Pipelines

Investiga qué es una pipeline en GitLab y qué ventajas estratégicas ofrece a los equipos de desarrollo.

Tu Resolución
Concepto de Pipeline CI/CD: Es el motor de Integración y Despliegue Continuo de GitLab. Consiste en una serie de flujos o fases automatizadas (compilar, testear, analizar calidad y desplegar) controladas a través de un archivo centralizado llamado .gitlab-ci.yml.

Ventajas operativas: Elimina los errores humanos derivados de pruebas manuales. Si un programador sube un cambio que rompe la lógica del negocio o no compila, la pipeline "falla" de inmediato bloqueando la integración y blindando el software corporativo contra fallos en producción.
Ejer 12

GitHub Actions y Secrets

Investiga GitHub Actions y explica el propósito fundamental de los Secrets dentro de la plataforma.

Tu Resolución
GitHub Actions: Es la alternativa nativa de GitHub para orquestar pipelines de automatización basadas en archivos YAML.

Utilidad esencial de los Secrets: Son variables de entorno cifradas de forma segura. En flujos automatizados de producción se requiere interactuar con servidores, bases de datos o servicios de la nube mediante contraseñas, tokens de API o claves SSH. Los Secrets permiten inyectar estos datos en los scripts de forma segura impidiendo que queden expuestos como texto plano en el historial visible del repositorio.

Reto

Reto final opcional

Describe un flujo de trabajo integral y profesional en el que intervengan: clonación, desarrollo en ramas, Merge Request y validación por pipeline automatizada.

Tu Resolución
Ecosistema de Trabajo Automatizado Extremo a Extremo:
  1. Fase Local: El desarrollador descarga el software con git clone, crea una rama aislada (feature/seguridad-tokens) y realiza sus commits. Al escribir, los hooks locales formatean el código y validan la sintaxis antes del commit.
  2. Publicación en GitLab: Se sube la rama mediante git push origin feature/... y se abre un Merge Request formal solicitando la revisión de los compañeros.
  3. Validación Autómata: GitLab capta el evento y dispara de inmediato una Pipeline de integración. Los servidores automáticos ejecutan los tests unitarios y comprueban los estándares de calidad de la empresa.
  4. Cierre del ciclo: El líder técnico del equipo revisa el reporte verde de la pipeline, aprueba visualmente el código del Merge Request y ejecuta el merge definitivo en main, disparando el despliegue automático del proyecto a producción.
