Ejer 1

Fusión de una rama sin conflicto

En la rama feature/footer, crea un archivo llamado footer.html y guarda los cambios con un commit. Después, vuelve a la rama principal y fusiona feature/footer. Comprueba el historial de commits y explica qué ha ocurrido tras hacer el merge.

Solución

En la rama master no existe el fichero

- Como master no tenía cambios nuevos que chocaran con la otra rama, Git no necesitó crear un commit de fusión especial. Simplemente "movió el puntero" de master hacia adelante hasta alcanzar el commit 32b2abb de la rama feature/footer.

- Ahora ambas ramas apuntan exactamente al mismo último commit, integrando el archivo footer.html en master de forma limpia y lineal.

Ejer 7

Caso práctico completo

Imagina que estás trabajando en equipo y dos personas han editado la misma parte del archivo index.html: una ha cambiado el texto del encabezado y otra ha cambiado ese mismo texto en otra rama distinta. Describe paso a paso cómo resolverías el conflicto, indicando qué comandos usarías desde el momento en que Git detecta el conflicto hasta que la fusión queda completada.

Solución

Paso Comando en la Terminal Qué hace el desarrollador / Qué ocurre
1. Detectar git merge rama-compañero Git detiene la fusión automáticamente y avisa en consola: CONFLICT (content): Merge conflict in index.html.
2. Verificar git status Permite comprobar qué archivos están en conflicto. Aparecerán en rojo bajo la etiqueta both modified: index.html.
3. Resolver Manual (en VS Code) Se abre index.html. El desarrollador limpia el código borrando las marcas <<<<<<<, =======, >>>>>>> y combinando los dos encabezados en uno final coherente.
4. Preparar git add index.html Se añade el archivo al Staging Area. Esto le indica a Git que el conflicto de ese archivo ya ha sido solucionado.
5. Concluir git commit -m "Fix: conflicto de encabezado resuelto" Se genera el commit de fusión (Merge commit). El repositorio vuelve a estar limpio y la terminal sale del estado (master|MERGING).
Ejer 8

Pull vs Fetch

Explica con tus palabras qué diferencia existe entre git pull y git fetch. Después, responde en qué situación usarías cada uno y qué ventaja tiene revisar los cambios antes de integrarlos en tu rama local.

Solución

Concepto / Comando git fetch git pull
¿Qué hace? Solo descarga los cambios del servidor remoto a tu base de datos local, pero no los aplica en tus archivos de trabajo. Descarga y fusiona los cambios del servidor remoto directamente en tu rama actual en un solo paso.
Impacto en tu código Seguro. No altera tus archivos ni provoca conflictos inmediatos. Riesgoso. Puede modificar tus archivos y generar conflictos de inmediato si hay código solapado.
Fórmula equivalente git fetch git fetch + git merge
💡 Ventaja de revisar los cambios antes de integrarlos:
Usar git fetch te permite evaluar qué ha hecho tu equipo (usando git log origin/master) antes de mezclarlo con tu código. Esto evita sorpresas, te permite planificar la resolución de posibles conflictos y garantiza que no rompas tu entorno de trabajo local de forma inesperada.
Ejer 9

Actualizar un repositorio remoto

Clona un repositorio de prueba o utiliza uno que ya tengas. Después, ejecuta el comando necesario para traer los cambios del repositorio remoto sin fusionarlos automáticamente. A continuación, consulta el estado del repositorio y el historial para comprobar qué información nueva ha llegado.

Solución

Paso Comando en la Terminal Explicación de lo que ocurre
1. Descargar git fetch origin Trae toda la información nueva y las ramas del servidor de GitHub a tu máquina, pero no modifica tus archivos de trabajo. Su espacio de trabajo sigue intacto.
2. Verificar Estado git status Git te informará exactamente cuántos commits por detrás estás respecto al servidor con un mensaje estilo: Your branch is behind 'origin/master' by X commits.
3. Revisar Cambios git log master..origin/master --oneline Te muestra la lista de commits específicos que están en GitHub pero que tú todavía no has fusionado en tu rama local, permitiéndote auditar el código antes de integrarlo.
Ejer 10

Git Stash

Realiza cambios en uno o varios archivos de tu repositorio sin hacer commit. Después, guarda temporalmente esos cambios con git stash. Comprueba que el directorio de trabajo queda limpio y, a continuación, recupera los cambios guardados. Explica para qué puede resultar útil esta herramienta en un entorno colaborativo.

Solución

Acción Comando en la Terminal ¿Qué ocurre en el repositorio?
1. Guardar temporalmente git stash Toma tus modificaciones actuales (no guardadas en commit) y las oculta en una pila temporal. Tu directorio de trabajo queda completamente limpio (igual al último commit).
2. Listar el baúl git stash list Muestra el historial de elementos que tienes guardados en el stash bajo identificadores como stash@{0}.
3. Recuperar y limpiar git stash pop Devuelve los últimos cambios guardados a tus archivos de trabajo y los elimina automáticamente de la lista temporal del stash.
💡 ¿Para qué es útil en un entorno colaborativo?
Es indispensable cuando estás programando una nueva funcionalidad a medias y surge una urgencia (un fallo en producción o un compañero que necesita revisión de su código). En lugar de hacer un commit incompleto o perder tu progreso, haces git stash, cambias de rama para resolver la urgencia, regresas a tu rama y usas git stash pop para continuar exactamente donde lo dejaste.
Ejer 11

Rebase

Crea una rama nueva a partir de master, realiza al menos un commit en ella y después actualiza esa rama utilizando git rebase sobre master. Explica qué hace exactamente este comando y qué diferencia principal existe entre usar merge y usar rebase.

Solución

Comando ¿Qué hace exactamente? Resultado en el historial
git rebase master Toma los commits que hiciste en tu rama secundaria, los "desengancha" temporalmente, añade los nuevos commits que se hayan hecho en master y luego vuelve a aplicar tus commits justo encima. Reescribe el historial para que parezca que creaste tu rama a partir del último commit de master, logrando una estructura completamente lineal y limpia.
Estrategia Ventajas Desventajas
git merge Es seguro y no altera el pasado; mantiene la historia real de cuándo y cómo se hicieron las cosas de forma exacta. El historial puede volverse caótico, lleno de ramificaciones entrelazadas y commits de fusión innecesarios ("grafiti de Git").
git rebase Deja un historial lineal, limpio y extremadamente fácil de leer o auditar en proyectos grandes. Peligroso si se usa en ramas públicas compartidas, ya que reescribe los identificadores (SHA-1) de los commits existentes.
